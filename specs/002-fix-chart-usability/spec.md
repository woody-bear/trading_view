# Feature Specification: 종목 상세화면 차트 오류 수정 및 사용성 개선

**Feature Branch**: `002-fix-chart-usability`
**Created**: 2026-03-20
**Status**: Draft
**Input**: User description: "종목 상세화면에 차트오류를 없애고 사용성을 높이는 작업"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 차트 데이터 없을 때 명확한 안내 (Priority: P1)

사용자가 상장폐지되거나 데이터가 없는 종목의 상세화면에 진입했을 때, 빈 차트 3개(메인/RSI/MACD)가 아무 설명 없이 표시되는 대신 "차트 데이터를 불러올 수 없습니다" 같은 명확한 안내 메시지를 보여준다.

**Why this priority**: 빈 차트는 사용자에게 시스템 오류로 인식되며, 가장 빈번하게 발생하는 혼란 요소이다.

**Independent Test**: 존재하지 않는 심볼(예: `INVALID123`)로 상세화면 접근 시 안내 메시지가 표시되는지 확인

**Acceptance Scenarios**:

1. **Given** 데이터가 없는 종목 심볼로 접근, **When** 차트 API가 빈 캔들 배열을 반환, **Then** 빈 차트 대신 "차트 데이터가 없습니다" 안내 메시지 표시
2. **Given** 차트 데이터 로딩 중, **When** 10초 이상 응답이 없을 때, **Then** 타임아웃 안내와 재시도 버튼 표시
3. **Given** 정상 데이터가 있는 종목, **When** 상세화면 진입, **Then** 기존과 동일하게 차트가 정상 렌더링

---

### User Story 2 - 차트 로딩 중 부분 콘텐츠 표시 (Priority: P1)

차트 데이터를 불러오는 동안 전체 페이지가 "로딩 중..." 텍스트로 막히는 대신, 종목명/현재가/지표 패널 등 이미 확보된 정보를 먼저 보여주고 차트 영역만 스켈레톤 UI로 표시한다.

**Why this priority**: 현재 차트 로딩이 전체 페이지를 차단하여 느린 네트워크에서 사용자가 빈 화면만 보게 된다.

**Independent Test**: 네트워크 속도를 제한(3G 시뮬레이션)한 상태에서 상세화면 진입 시 종목 정보가 차트보다 먼저 표시되는지 확인

**Acceptance Scenarios**:

1. **Given** 상세화면 진입, **When** 차트 데이터 로딩 중, **Then** 종목명/현재가/신호 상태는 즉시 표시되고 차트 영역은 스켈레톤 UI 표시
2. **Given** 스켈레톤 상태에서, **When** 차트 데이터 도착, **Then** 스켈레톤이 실제 차트로 부드럽게 전환

---

### User Story 3 - API 실패 시 사용자 피드백 (Priority: P2)

민감도 변경, 관심종목 추가 등 사용자 액션이 실패했을 때 아무런 피드백 없이 무시되는 대신, 토스트 메시지로 성공/실패를 알려준다.

**Why this priority**: 현재 민감도 변경의 catch 블록이 비어 있어 사용자가 설정 실패를 인지할 수 없다.

**Independent Test**: 백엔드를 중지한 상태에서 민감도 변경 시 에러 토스트가 표시되는지 확인

**Acceptance Scenarios**:

1. **Given** 민감도 변경 시도, **When** API 호출 실패, **Then** "설정 변경에 실패했습니다" 토스트 메시지 표시
2. **Given** 관심종목 추가 시도, **When** 성공, **Then** "관심종목에 추가되었습니다" 확인 메시지 표시
3. **Given** 관심종목 추가 시도, **When** 이미 등록된 종목, **Then** "이미 등록된 종목입니다" 안내 메시지 표시

---

### User Story 4 - 실시간 연결 상태 가시성 (Priority: P2)

SSE 실시간 가격 연결이 끊겼을 때 녹색 "실시간" 인디케이터가 사라지기만 하는 대신, 연결 상태(연결됨/재연결 중/연결 끊김)를 명확히 표시한다.

**Why this priority**: 실시간 데이터가 갱신되지 않는 상황에서 사용자가 이를 인지하지 못하면 오래된 가격으로 잘못된 판단을 내릴 수 있다.

**Independent Test**: 네트워크를 일시 차단했다가 복구했을 때 연결 상태 인디케이터가 올바르게 전환되는지 확인

**Acceptance Scenarios**:

1. **Given** SSE 연결 정상, **When** 가격 데이터 수신 중, **Then** "실시간" 인디케이터 표시 (녹색)
2. **Given** SSE 연결 끊김, **When** 자동 재연결 시도 중, **Then** "재연결 중..." 인디케이터 표시 (황색)
3. **Given** 재연결 실패, **When** 연속 3회 재연결 실패 (약 30초 경과), **Then** "연결 끊김" 인디케이터와 수동 재연결 버튼 표시

---

### User Story 5 - 차트 시그널 마커 렌더링 안정성 (Priority: P2)

BUY/SELL 시그널 마커가 데이터 형식 오류로 렌더링에 실패할 때 아무 표시 없이 무시되는 대신, 마커 없이 차트를 정상 표시하고 마커 미표시 안내를 보여준다.

**Why this priority**: 마커 렌더링 실패가 조용히 무시되어 사용자가 시그널 정보가 없는 차트를 정상으로 오인할 수 있다.

**Independent Test**: 잘못된 형식의 마커 데이터를 반환하도록 API를 모킹한 후 차트가 정상 표시되고 경고가 나타나는지 확인

**Acceptance Scenarios**:

1. **Given** 마커 데이터가 유효하지 않을 때, **When** 차트 렌더링, **Then** 캔들차트/지표는 정상 표시되고 "시그널 마커를 표시할 수 없습니다" 안내 표시
2. **Given** 마커 데이터가 정상일 때, **When** 차트 렌더링, **Then** BUY/SELL 마커가 정상 표시

---

### User Story 6 - 차트 컴포넌트 에러 격리 (Priority: P3)

차트 렌더링 중 예기치 못한 오류가 발생해도 전체 상세 페이지가 크래시되지 않고, 차트 영역만 에러 상태로 표시하며 나머지 정보(가격, 지표, 설정)는 계속 사용 가능하다.

**Why this priority**: 차트 라이브러리 오류가 전체 앱을 크래시시키면 사용자가 어떤 정보에도 접근할 수 없게 된다.

**Independent Test**: 차트 컴포넌트에 의도적으로 오류를 발생시킨 후 페이지 나머지 부분이 정상 동작하는지 확인

**Acceptance Scenarios**:

1. **Given** 차트 렌더링 중 오류 발생, **When** 에러 바운더리가 포착, **Then** 차트 영역에 "차트를 표시할 수 없습니다. 새로고침해 주세요." 안내 표시
2. **Given** 차트 영역이 에러 상태, **When** 사용자가 다른 기능 사용, **Then** 가격 정보, 지표 패널, 관심종목 추가 등은 정상 동작

---

### User Story 7 - 마지막 일봉 이상값 방지 (Priority: P1)

yfinance에서 당일 미완성 캔들(장중 데이터)이 캐시되면서 마지막 일봉이 급락/급등하는 것처럼 보이는 현상을 해결한다. 또한 실시간 가격 업데이트가 캔들 OHLC를 잘못 덮어쓰는 문제도 함께 수정한다.

**Why this priority**: 차트의 마지막 캔들이 비정상적으로 뚝 떨어지거나 치솟으면 사용자가 잘못된 매매 판단을 내릴 수 있다. 가격 데이터 정확성은 차트 시스템의 핵심 신뢰도와 직결된다.

**Independent Test**: 장중에 한국 종목(예: 005930) 차트를 조회하여 마지막 캔들이 전일 종가 대비 비정상적 갭 없이 자연스럽게 이어지는지 확인

**Acceptance Scenarios**:

1. **Given** 장중(한국 15:30 전, 미국 16:00 전)에 차트 조회, **When** yfinance가 당일 미완성 캔들을 포함하여 반환, **Then** 미완성 캔들은 제거하고 전일까지의 완성된 캔들만 표시하되, 실시간 가격이 있으면 당일 캔들을 실시간 데이터로 구성하여 표시. 실시간 가격이 아직 수신되지 않은 경우에는 완성 캔들까지만 표시 (빈 당일 캔들을 만들지 않음)
2. **Given** SSE 실시간 가격 수신 중, **When** 마지막 캔들에 가격 반영, **Then** open은 캔들 원본 값을 유지하고 high/low/close만 업데이트
3. **Given** yfinance auto_adjust=True로 수정주가 반환, **When** KIS 실시간 가격(비수정)과 혼합, **Then** 가격 불일치가 발생하지 않도록 당일 캔들은 실시간 가격 기준으로 독립 구성
4. **Given** 장 마감 후 차트 조회, **When** 캐시에 완성된 당일 캔들이 있음, **Then** 추가 다운로드 없이 정상 캔들 표시

---

### User Story 8 - 캐시 신선도 시간대 인식 (Priority: P2)

차트 캐시 신선도 판단 시 UTC 기준 20시간이 아닌, 각 시장의 장 마감 시간을 기준으로 판단하여 불필요한 재다운로드를 방지하고 완성된 캔들만 캐시한다.

**Why this priority**: 현재 UTC 기준 20시간 캐시 정책은 KR/US 시장 시간대를 고려하지 않아, 장 마감 후에도 미완성 캔들이 캐시에 남거나, 반대로 유효한 캐시를 불필요하게 갱신한다.

**Independent Test**: KR 시장 장 마감(15:30 KST) 후 16시에 조회하면 당일 완성 캔들이 캐시되고, 다음 날 9시 전까지 재다운로드 없이 캐시가 유효한지 확인

**Acceptance Scenarios**:

1. **Given** KR 종목, 15:30 KST 이후 조회, **When** 캐시의 마지막 캔들이 당일 날짜, **Then** 캐시를 fresh로 판단하고 재다운로드하지 않음
2. **Given** US 종목, 16:00 ET 이후 조회, **When** 캐시의 마지막 캔들이 당일 날짜, **Then** 캐시를 fresh로 판단
3. **Given** KR 종목, 다음 영업일 09:00 KST 이후 조회, **When** 캐시의 마지막 캔들이 전일, **Then** 캐시를 stale로 판단하고 증분 다운로드

---

### User Story 9 - BUY/SELL 마커 호버 시 색상 강조 (Priority: P3)

차트의 BUY/SELL 시그널 마커 위에 마우스를 올렸을 때 마커 텍스트 색상이 변경되어 시각적 피드백을 제공한다. lightweight-charts v5의 `subscribeCrosshairMove()` + 마커 `id` 필드 + `setMarkers()` 재호출 방식으로 구현한다.

**Why this priority**: 핵심 기능이 아닌 UX 개선이므로 P3. 차트 데이터 정확성과 에러 처리가 우선이다.

**Independent Test**: BUY/SELL 마커가 있는 차트에서 마커 위치에 크로스헤어를 이동했을 때 해당 마커의 색상이 변경되는지 확인

**Acceptance Scenarios**:

1. **Given** BUY/SELL 마커가 표시된 차트 (PC), **When** 크로스헤어가 마커와 같은 시간 위치에 도달, **Then** 해당 마커의 색상이 강조색으로 변경 (BUY: `#22c55e`→`#4ade80`, SELL: `#ef4444`→`#f87171`)
2. **Given** 마커가 강조 상태, **When** 크로스헤어가 마커 위치를 벗어남, **Then** 마커 색상이 원래 색으로 복원
3. **Given** 모바일 환경 (`hover: hover` 미지원), **When** 터치 기기에서 차트 표시, **Then** 호버 이벤트 리스너를 등록하지 않음 (성능 보호). 단, 마커 탭(클릭)에 의한 매수지점 기록(US10)은 모바일에서도 정상 동작

---

### User Story 10 - BUY 마커 클릭 시 매수지점 기록 (Priority: P3)

BUY 시그널 마커를 클릭하면 해당 시점을 "내 매수지점"으로 차트 위에 직관적으로 기록한다. 기록된 매수지점은 가격 수평선과 라벨로 표시되어, 현재가 대비 수익률을 한눈에 파악할 수 있다.

**Why this priority**: 매매 기록은 부가 기능이며, 차트 데이터 정확성/에러 처리(P1~P2)가 우선이다.

**Independent Test**: BUY 마커를 클릭한 후 차트에 매수 가격 수평선이 표시되고, 페이지 새로고침 후에도 유지되는지 확인

**Acceptance Scenarios**:

1. **Given** BUY 마커가 표시된 차트, **When** 사용자가 BUY 마커를 클릭 (PC: `subscribeClick`, 모바일: 탭), **Then** 해당 시점의 가격이 "매수지점"으로 기록되고 차트에 가격 수평선 + 시장에 맞는 통화 라벨 표시 (KR: "매수 ₩50,000", US: "매수 $175.50", CRYPTO: "매수 $45,200")
2. **Given** 매수지점이 기록된 상태, **When** 실시간 가격이 업데이트됨, **Then** 매수가 대비 현재가 수익률이 라벨에 함께 표시 (소수점 1자리, 부호 포함: "+2.5%" 또는 "-1.3%")
3. **Given** 매수지점이 기록된 상태, **When** 사용자가 같은 BUY 마커를 다시 클릭, **Then** 매수지점 기록이 삭제됨 (토글). 다른 BUY 마커를 클릭하면 기존 기록이 대체됨
4. **Given** 매수지점이 기록된 상태, **When** 페이지를 새로고침하거나 재방문, **Then** localStorage에서 복원되어 기록이 유지됨. localStorage가 불가한 환경(프라이빗 브라우징 등)에서는 세션 내에서만 유지

### Edge Cases

- 차트 데이터가 20개 미만의 캔들만 있을 때 볼린저밴드/RSI가 계산 불가한 경우 → 가용한 지표만 표시하고 미계산 지표에 대한 안내 제공
- 빠른 페이지 전환(종목 A → 종목 B) 시 이전 종목의 SSE 연결이 정리되지 않는 경우 → 이전 연결 정리 후 새 연결 수립
- 모바일에서 빠른 실시간 업데이트(초당 1~2회)로 CPU 과부하 → 업데이트 빈도 조절(throttle)
- yfinance가 주말/공휴일에 미완성 캔들을 반환하는 경우 → 시장별 영업일 기준으로 캔들 완성 여부 판단
- 주식 분할(stock split) 직후 yfinance 수정주가와 실시간 가격 불일치 → 당일 캔들은 실시간 가격 기준으로 독립 구성
- 암호화폐(24시간 거래)의 경우 "장 마감" 개념이 없음 → UTC 00:00 기준 일봉 마감으로 처리
- localStorage 사용 불가 환경(프라이빗 브라우징, 스토리지 가득 참)에서 매수지점 저장 실패 → 세션 메모리에서만 유지, 저장 실패 시 토스트로 안내
- DST(서머타임) 전환일(미국 3월/11월, 연 2회)에 US 시장 마감 시간이 1시간 변동 → `America/New_York` zoneinfo가 자동 처리

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 차트 API가 빈 캔들 배열을 반환할 경우 시스템은 빈 차트 대신 안내 메시지를 표시해야 한다
- **FR-002**: 차트 데이터 로딩 중 종목명, 현재가, 신호 상태 등 이미 확보된 정보를 먼저 표시하고 차트 영역은 스켈레톤 UI로 표시해야 한다
- **FR-003**: 차트 데이터 요청이 10초 내 응답하지 않으면 타임아웃 안내와 재시도 옵션을 제공해야 한다
- **FR-004**: 사용자 액션(민감도 변경, 관심종목 추가/삭제) 성공 및 실패 시 토스트 메시지로 피드백을 제공해야 한다
- **FR-005**: SSE 실시간 연결 상태를 "연결됨/재연결 중/연결 끊김" 3단계로 시각적으로 표시해야 한다
- **FR-006**: 시그널 마커 렌더링 실패 시 차트 본체는 정상 표시하고 마커 미표시 안내를 제공해야 한다
- **FR-007**: 차트 컴포넌트에서 발생하는 런타임 오류가 전체 페이지를 크래시시키지 않도록 에러 바운더리로 격리해야 한다
- **FR-008**: 페이지 전환 시 이전 종목의 SSE 연결 및 차트 리소스를 정리(cleanup)해야 한다
- **FR-009**: 모바일 환경에서 실시간 차트 업데이트 빈도를 적절히 조절하여 과도한 렌더링을 방지해야 한다
- **FR-010**: 장중 yfinance 데이터의 미완성 당일 캔들을 감지하여 제거하고, 실시간 가격이 있으면 당일 캔들을 실시간 데이터로 독립 구성해야 한다
- **FR-011**: 차트 캐시 신선도 판단 시 시장별 장 마감 시간(KR: 15:30 KST, US: 16:00 ET, CRYPTO: UTC 00:00)을 기준으로 완성 캔들 여부를 판단해야 한다
- **FR-012**: 실시간 가격으로 마지막 캔들 업데이트 시 원래 캔들의 open 값을 보존하고 high/low/close만 갱신해야 한다
- **FR-013**: BUY/SELL 시그널 마커 위에 크로스헤어가 위치하면 해당 마커의 색상을 강조색으로 변경하고 (BUY: `#22c55e`→`#4ade80`, SELL: `#ef4444`→`#f87171`), 벗어나면 원래 색으로 복원해야 한다 (PC 전용 — `hover: hover` 미디어 쿼리로 판단, 모바일에서는 호버 리스너 비등록)
- **FR-014**: BUY 마커 클릭(PC/모바일 모두) 시 해당 시점의 가격을 매수지점으로 기록하고, 차트에 가격 수평선 + 시장별 통화 포맷 수익률 라벨(소수점 1자리, 부호 포함)로 시각화해야 한다
- **FR-015**: 기록된 매수지점은 localStorage에 저장되어 페이지 새로고침/재방문 시에도 유지되어야 한다 (키: `buyPoints:{symbol}`)
- **FR-016**: 같은 BUY 마커를 재클릭하면 매수지점 기록이 삭제(토글)되고, 다른 BUY 마커를 클릭하면 기존 기록이 대체되어야 한다

### Key Entities

- **ChartData**: 캔들(OHLCV), 기술적 지표(BB/RSI/MACD/EMA), 시그널 마커, 스퀴즈 도트를 포함하는 차트 표시 데이터
- **ConnectionStatus**: SSE/WebSocket 실시간 연결의 현재 상태 (connected / reconnecting / disconnected)
- **UserFeedback**: 사용자 액션 결과를 전달하는 토스트 메시지 (성공/실패/정보)
- **BuyPoint**: 사용자가 기록한 매수지점 (symbol, price, date, markerTime). localStorage에 종목별 1개만 저장. 새 클릭 시 이전 기록 대체

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 데이터가 없는 종목 진입 시 100% 안내 메시지가 표시되며, 빈 차트가 노출되는 경우가 0건이어야 한다
- **SC-002**: 상세화면 진입 후 종목 기본 정보(이름, 가격)가 1초 이내에 표시되어야 한다
- **SC-003**: 모든 사용자 액션(민감도 변경, 관심종목 추가)에 대해 2초 이내에 성공/실패 피드백이 제공되어야 한다
- **SC-004**: SSE 연결 끊김 시 3초 이내에 사용자에게 상태 변경이 시각적으로 전달되어야 한다
- **SC-005**: 차트 컴포넌트 오류 발생 시 페이지 나머지 기능의 정상 동작률이 100%이어야 한다
- **SC-006**: 모바일 환경에서 실시간 업데이트 중 60fps 이상 유지되어야 하며, Chrome DevTools Performance 패널 기준 100ms 이상의 long task가 발생하지 않아야 한다
- **SC-007**: 장중 조회 시 마지막 캔들이 전일 종가 대비 비정상적 갭을 보이지 않아야 한다. 단, 실적 발표/시간외 뉴스 등 정상적인 갭업/갭다운은 허용하며, 시스템 오류에 의한 인위적 갭(yfinance 미완성 캔들, 수정주가 혼합)이 0건이어야 한다
- **SC-008**: 장 마감 후 차트 캐시가 다음 영업일 장 시작 전까지 유효하게 유지되어야 한다

## Clarifications

### Session 2026-03-21

- Q: BUY/SELL 마커에 마우스 호버 시 색상 변경이 기술적으로 가능한가? → A: 가능. lightweight-charts v5의 `subscribeCrosshairMove()` + 마커 `id` + `setMarkers()` 재호출로 구현. User Story 9, FR-013으로 추가 완료.
- Q: 매수지점 데이터를 어디에 저장할까? → A: localStorage (브라우저 로컬 저장, 백엔드 변경 없음, 기기 간 공유 불가)
- Q: 종목당 매수지점 최대 개수? → A: 1개 (마지막 클릭이 이전 기록을 대체)

## Assumptions

- 토스트 메시지 컴포넌트는 기존 UI 라이브러리를 활용하거나 간단히 구현 가능하다
- 차트 데이터 타임아웃 기준은 10초로 설정한다
- SSE 재연결 로직은 기존 EventSource의 자동 재연결을 활용하되 UI 상태만 추가한다
- 모바일 throttle 기준은 200ms 간격으로 설정한다 (초당 최대 5회 업데이트)
