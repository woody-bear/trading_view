# Feature Specification: 장 마감 후 차트 데이터 정상화 및 로드 방식 개선

**Feature Branch**: `004-fix-chart-after-close`
**Created**: 2026-03-22
**Status**: Draft
**Input**: User description: "장이 종료된후에 한국, 미국종목의 차트의 데이터가 정상적으로 안나오는 현상 및 차트로드 방식개선"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 장 마감 후 차트 데이터 정상 표시 (Priority: P1)

사용자가 장 마감 후(한국 15:30 이후, 미국 16:00 ET 이후) 또는 주말에 종목 상세화면에 진입했을 때, 차트 데이터가 정상적으로 표시되어야 한다. 현재 장 마감 후에 차트가 빈 화면으로 나오거나, 캔들이 비정상적으로 적게 표시되거나, 로딩이 무한히 지속되는 현상을 해결한다.

**Why this priority**: 장 마감 후에도 차트를 확인하여 다음 날 매매 전략을 세우는 것이 사용자의 핵심 사용 패턴이다.

**Independent Test**: 장 마감 후 또는 주말에 한국/미국 종목의 상세화면에 진입하여 차트가 200개 캔들로 정상 렌더링되는지 확인

**Acceptance Scenarios**:

1. **Given** 한국 장 마감 후(15:30 KST 이후), **When** KR 종목 상세화면에 진입, **Then** 당일까지 포함한 완성 캔들이 정상 표시됨
2. **Given** 미국 장 마감 후(16:00 ET 이후), **When** US 종목 상세화면에 진입, **Then** 당일까지 포함한 완성 캔들이 정상 표시됨
3. **Given** 주말(토/일), **When** 아무 종목 상세화면에 진입, **Then** 금요일까지의 캔들이 정상 표시됨
4. **Given** 캐시가 없는 신규 종목, **When** 장 마감 후 최초 조회, **Then** 전체 히스토리가 다운로드되어 정상 표시됨

---

### User Story 2 - 차트 로딩 속도 개선 (Priority: P1)

차트 데이터 조회 시 캐시가 유효한 경우 즉시 반환하고, 캐시가 없거나 오래된 경우에만 외부 데이터를 다운로드한다. 모든 시나리오에서 차트 로딩이 5초 이내에 완료되어야 한다.

**Why this priority**: 느린 차트 로딩은 사용자 이탈의 직접적 원인이다.

**Independent Test**: 캐시 유효 시 차트 로딩 시간이 1초 이내인지, 캐시 없을 때 5초 이내인지 측정

**Acceptance Scenarios**:

1. **Given** 유효한 캐시가 존재할 때, **When** 차트를 조회, **Then** 1초 이내에 차트가 표시됨
2. **Given** 캐시가 없을 때, **When** 차트를 최초 조회, **Then** 5초 이내에 차트가 표시됨
3. **Given** 캐시가 오래되었을 때 (다음 영업일), **When** 차트를 조회, **Then** 새 데이터를 다운로드하여 5초 이내에 표시됨
4. **Given** 외부 데이터 소스가 일시적으로 불가할 때, **When** 차트를 조회, **Then** 기존 캐시 데이터라도 표시하고 사용자에게 "최신 데이터가 아닐 수 있습니다" 안내

---

### User Story 3 - 차트 로딩 상태 가시성 (Priority: P2)

차트 데이터 로딩 중 사용자에게 현재 상태를 명확히 알려준다. 특히 외부 데이터 다운로드가 필요한 경우 "데이터 다운로드 중..." 메시지를 표시하여 사용자가 빈 화면에서 기다리지 않게 한다.

**Why this priority**: 로딩 시간이 긴 경우 사용자에게 진행 상태를 보여주는 것은 UX의 기본이나, 핵심 기능(데이터 정상 표시)이 우선이다.

**Independent Test**: 캐시 미스 시 "데이터 다운로드 중..." 메시지가 스켈레톤 위에 표시되는지 확인

**Acceptance Scenarios**:

1. **Given** 캐시 HIT, **When** 차트 로딩, **Then** 스켈레톤 → 차트 즉시 전환 (추가 메시지 없음)
2. **Given** 캐시 MISS (다운로드 필요), **When** 차트 로딩, **Then** "차트 데이터 다운로드 중..." 메시지가 스켈레톤 위에 표시
3. **Given** 다운로드 실패 후 기존 캐시 사용, **When** 차트 표시, **Then** "최신 데이터가 아닐 수 있습니다" 안내 배너 표시

---

### Edge Cases

- 한국 장 마감 직후(15:30~15:35 KST) yfinance 데이터가 아직 반영되지 않은 경우 → 기존 캐시를 반환하고, 다음 조회 시 갱신
- 미국 장 마감 시간이 DST(서머타임)에 따라 변동 → 시간대 라이브러리로 자동 처리
- 한국 공휴일 (장 미개장일) → 직전 영업일 데이터가 최신으로 판단
- 네트워크 오류로 yfinance 다운로드 실패 → 기존 캐시 반환 + 에러 로깅
- 타임프레임 전환(일봉↔주봉) 시 각 타임프레임별 독립 캐시 → 전환 시 해당 타임프레임 캐시 조회

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 장 마감 후 차트 조회 시 당일 완성 캔들을 포함한 전체 히스토리가 정상 반환되어야 한다
- **FR-002**: 주말/공휴일 차트 조회 시 마지막 영업일까지의 완성 캔들이 정상 반환되어야 한다
- **FR-003**: 캐시가 유효한 경우 외부 데이터 소스에 접근하지 않고 캐시에서 즉시 반환해야 한다
- **FR-004**: 캐시가 오래된 경우 새 데이터를 다운로드하되, 실패 시 기존 캐시를 반환해야 한다
- **FR-005**: 차트 캐시 유효성 판단은 시장별 장 마감 시간을 기준으로 해야 한다 (KR: 15:30 KST, US: 16:00 ET)
- **FR-006**: 캐시 데이터의 무결성 검증 — 캔들 수가 50개 미만이거나 가격이 비정상인 경우 캐시를 폐기하고 재다운로드해야 한다
- **FR-007**: 캐시 미스 시 프론트엔드에 "데이터 다운로드 중..." 상태를 전달해야 한다
- **FR-008**: 외부 데이터 다운로드 실패 후 기존 캐시 사용 시 "최신 데이터가 아닐 수 있습니다" 안내를 표시해야 한다
- **FR-009**: 일봉, 주봉, 월봉 각 타임프레임별로 독립적인 캐시를 유지해야 한다

### Key Entities

- **ChartCache**: 종목별/타임프레임별 OHLCV 캔들 데이터 캐시 — 심볼, 시장, 타임프레임, 캔들 배열, 최종 갱신 시각
- **CacheFreshness**: 캐시 유효성 판단 결과 — fresh (즉시 반환) / stale (재다운로드) / fallback (실패 시 기존 캐시 사용)

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 장 마감 후 또는 주말에 한국/미국 종목 차트 조회 시 100% 정상 데이터가 표시되어야 한다
- **SC-002**: 캐시 유효 시 차트 로딩 시간이 1초 이내여야 한다
- **SC-003**: 캐시 미스 시 차트 로딩 시간이 5초 이내여야 한다
- **SC-004**: 외부 데이터 다운로드 실패 시에도 기존 캐시 데이터가 표시되어야 한다 (빈 차트 0건)
- **SC-005**: 캐시 무결성 실패(50개 미만 캔들) 시 자동 재다운로드가 수행되어야 한다

## Assumptions

- 차트 캐시는 parquet 파일로 저장되며, 종목/시장/타임프레임 조합별 1개 파일이다 (기존 구현)
- yfinance가 주요 데이터 소스이며, 장 마감 후 수 분 이내에 당일 데이터가 반영된다
- 서버 시간대는 KST(Asia/Seoul)이며, 미국 시장은 America/New_York 시간대를 사용한다
- 캐시 유효성 판단의 "영업일"은 주말(토/일)만 제외하며, 한국 공휴일은 미처리한다 (간단 구현)
- 기존 스켈레톤 UI(ChartSkeleton)와 빈 상태 안내(ChartEmptyState)는 002 피처에서 구현 완료
