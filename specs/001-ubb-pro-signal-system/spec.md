# Feature Specification: UBB Pro Signal System

**Feature Branch**: `001-ubb-pro-signal-system`
**Created**: 2026-03-16
**Status**: Draft
**Input**: TradingView 기술적 지표 기반 한국/미국/비트코인 종목 실시간 모니터링 + 매매 신호 전환 시 텔레그램 자동 알림 로컬 시스템

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 자동 매매 신호 알림 수신 (Priority: P1)

개인 투자자가 관심 종목의 기술적 지표 조합(볼린저밴드, RSI, MACD, EMA, 거래량)을 기반으로 매수/매도 신호가 발생하면 텔레그램 메시지로 즉시 알림을 받는다. 사용자는 차트를 계속 지켜보지 않아도 중요한 매매 시점을 놓치지 않는다.

**Why this priority**: 시스템의 핵심 가치. 기술적 지표 분석 → 신호 판정 → 알림 발송이라는 전체 파이프라인이 동작해야 시스템 존재 의미가 있다. 이것만 동작해도 CLI 기반 MVP로 즉시 사용 가능하다.

**Independent Test**: 하드코딩된 워치리스트 1개 종목으로 시세 데이터 수집 → 지표 계산 → 신호 판정 → 텔레그램 발송까지 전체 흐름을 테스트할 수 있다.

**Acceptance Scenarios**:

1. **Given** 워치리스트에 종목이 등록되어 있고 시스템이 실행 중일 때, **When** 해당 종목의 기술적 지표 조합이 매수 조건(BB 하단 터치, RSI < 30, MACD 상승 전환, 거래량 > 평균 1.2배)을 모두 충족하면, **Then** 텔레그램으로 매수 신호 알림(종목명, 현재가, 신호 방향, 강도 점수, 지표 상세)이 발송된다.
2. **Given** 이전 스캔에서 신호 상태가 BUY였을 때, **When** 다음 스캔에서 신호 상태가 여전히 BUY이면, **Then** 중복 알림을 발송하지 않는다(상태 전환 시에만 발송).
3. **Given** 매수 신호가 발생한 지 30분 이내일 때, **When** 동일 종목에서 동일 방향 신호가 다시 감지되면, **Then** 쿨다운 정책에 따라 알림을 스킵한다.
4. **Given** 매수 상태인 종목에서, **When** RSI가 50 이상으로 회복되고 가격이 볼린저밴드 중간선을 상회하거나 MACD가 하락 전환하면, **Then** NEUTRAL 상태로 복귀하고 이를 기록한다.

### User Story 2 - 워치리스트 관리 (Priority: P2)

사용자가 한국 주식(종목번호), 미국 주식(티커), 비트코인(심볼)을 워치리스트에 추가/제거하고, 각 종목별로 분석 타임프레임(15분~일봉)을 설정한다. 시스템은 등록된 종목만 주기적으로 모니터링한다.

**Why this priority**: 신호 엔진이 어떤 종목을 감시할지 결정하는 데이터 관리 기능. P1의 하드코딩 워치리스트를 동적으로 관리 가능하게 만든다.

**Independent Test**: 워치리스트에 종목을 추가/삭제하고, 추가된 종목이 다음 스캔 주기에 포함되는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 빈 워치리스트가 있을 때, **When** 사용자가 한국 주식 종목번호(예: 005930)를 입력하면, **Then** 시스템이 해당 종목의 유효성을 검증하고 종목명(삼성전자)을 자동 표시하여 워치리스트에 추가한다.
2. **Given** 워치리스트에 종목이 있을 때, **When** 사용자가 해당 종목을 비활성화하면, **Then** 다음 스캔 주기부터 해당 종목이 모니터링에서 제외된다.
3. **Given** 미국 주식 종목을 추가할 때, **When** 존재하지 않는 티커를 입력하면, **Then** 유효성 검증 실패 메시지를 표시하고 추가를 거부한다.

### User Story 3 - 시장별 자동 스케줄링 (Priority: P3)

시스템이 시장별 거래 시간(한국 09:00~15:30 KST, 미국 09:30~16:00 ET, 비트코인 24시간)을 인식하여 10분 간격으로 장중 종목만 스캔한다. 사용자가 별도로 시작/중지할 필요 없이 시장 개장 시 자동으로 모니터링이 활성화된다.

**Why this priority**: P1/P2가 동작하면 수동 실행으로도 쓸 수 있지만, 자동 스케줄링이 있어야 "설정 후 잊기(set-and-forget)" 경험이 완성된다.

**Independent Test**: 시스템을 실행한 상태에서 한국 장중 시간에 한국 종목만 스캔되고, 장외 시간에는 스킵되는지 로그로 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 워치리스트에 한국/미국/비트코인 종목이 각각 있을 때, **When** 한국 장중 시간(KST 09:00~15:30 평일)이면, **Then** 한국 종목과 비트코인 종목만 스캔하고 미국 종목은 스킵한다.
2. **Given** 미국 서머타임이 적용 중일 때, **When** 미국 장중 시간을 판별하면, **Then** KST 22:30~05:00 기준으로 올바르게 스캔 범위를 결정한다.
3. **Given** 시스템이 10분 간격으로 스캔 중일 때, **When** 특정 종목의 데이터 수집이 실패하면, **Then** 해당 종목만 스킵하고 나머지 종목은 정상 스캔을 계속한다.

---

### User Story 4 - 웹 대시보드 실시간 현황 (Priority: P4)

사용자가 로컬 웹 브라우저에서 전체 워치리스트의 현재 신호 상태, 지표 값, 신호 이력을 한눈에 확인한다. 신호 전환 시 페이지 새로고침 없이 실시간으로 업데이트된다.

**Why this priority**: 텔레그램 알림만으로도 핵심 가치는 전달되지만, 대시보드가 있으면 전체 포트폴리오 상태를 시각적으로 파악할 수 있어 사용 편의성이 크게 향상된다.

**Independent Test**: 대시보드에 접속하여 종목 카드의 신호 상태, RSI/BB 게이지, 스퀴즈 단계를 확인하고, 신호 전환 시 실시간 업데이트되는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 시스템이 실행 중이고 워치리스트에 종목이 있을 때, **When** 사용자가 대시보드에 접속하면, **Then** 시장별로 그룹화된 종목 카드에 현재 신호 상태(BUY/SELL/NEUTRAL), 신호 강도(STRONG/NORMAL/WEAK), 현재가, 등락률, RSI, %B, 스퀴즈 단계가 표시된다.
2. **Given** 대시보드가 열려 있을 때, **When** 백엔드에서 특정 종목의 신호가 전환되면, **Then** 해당 종목 카드가 페이지 새로고침 없이 즉시 업데이트된다.
3. **Given** 종목 카드를 클릭할 때, **When** 신호 상세 뷰로 이동하면, **Then** 지표별 현재 값 상세, 신호 전환 이력 타임라인, 알림 발송 이력이 표시된다.
4. **Given** 종목 상세 뷰에 진입할 때, **When** 차트 영역이 렌더링되면, **Then** 캔들스틱 차트 위에 볼린저밴드(상단/중간/하단), EMA(20/50/200)가 오버레이되고, 하단에 RSI 서브차트, MACD 서브차트(히스토그램+시그널+MACD선), 거래량 바, 스퀴즈 도트가 TradingView와 동일한 시각적 형태로 표시된다.
5. **Given** 차트가 표시된 상태에서, **When** 신호 전환 시점이 존재하면, **Then** 해당 캔들 위치에 BUY/SELL 마커(화살표 또는 라벨)가 표시되어 신호 발생 지점을 시각적으로 확인할 수 있다.

---

### User Story 5 - 데이터 수집 안정성 (Priority: P5)

시스템이 1차 데이터 소스 장애 시 자동으로 대체 소스(fallback)로 전환하여 서비스 중단 없이 모니터링을 지속한다. 연속 실패 시 사용자에게 시스템 에러 알림을 발송한다.

**Why this priority**: 장기 운영 안정성을 보장하는 기능. 초기에는 단일 소스로도 동작하지만, 1인 무인 운영을 위해 fallback과 에러 알림이 필수적이다.

**Independent Test**: 1차 데이터 소스를 의도적으로 차단하고 fallback 소스로 자동 전환되는지, 모든 소스 실패 시 에러 알림이 발송되는지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 한국 주식 1차 데이터 소스가 응답하지 않을 때, **When** 스캔 주기가 도래하면, **Then** 자동으로 2차 소스에서 데이터를 가져와 지표 계산을 정상 수행한다.
2. **Given** 특정 종목의 모든 데이터 소스가 실패할 때, **When** 3회 연속 실패가 누적되면, **Then** 텔레그램으로 시스템 에러 알림을 발송한다.
3. **Given** 수집된 캔들 데이터가 지표 계산에 필요한 최소 개수(200개)에 미달할 때, **When** 지표 계산을 시도하면, **Then** 해당 종목을 스킵하고 경고 로그를 기록한다.

---

### User Story 6 - TradingView 웹훅 수신 (Priority: P6)

TradingView 유료 플랜 사용자가 Pine Script 지표의 Alert를 설정하면, 시스템이 웹훅을 직접 수신하여 로컬 지표 계산과 동일한 신호 파이프라인으로 처리한다. 로컬 계산과 웹훅 수신을 하이브리드로 운영할 수 있다.

**Why this priority**: 핵심 기능(로컬 지표 계산)이 이미 동작하므로, TradingView 연동은 정확도 향상을 위한 보완 채널이다. 무료 사용자는 이 기능 없이도 시스템을 완전히 활용할 수 있다.

**Independent Test**: 웹훅 엔드포인트에 TradingView Alert 형식의 테스트 페이로드를 전송하고, 시크릿 검증 → 신호 파이프라인 합류 → 알림 발송까지 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 시스템이 외부 터널(ngrok/cloudflared)을 통해 노출되어 있을 때, **When** TradingView에서 유효한 시크릿과 함께 Alert 웹훅을 전송하면, **Then** 시스템이 웹훅을 수신하고 신호 파이프라인에 합류시킨다.
2. **Given** 웹훅 요청이 도착할 때, **When** 시크릿 검증이 실패하면, **Then** 요청을 거부하고 보안 로그를 기록한다.

---

### Edge Cases

- 동일 종목에서 BUY → SELL 신호가 쿨다운 기간 내에 연속 발생하면? → 방향 전환은 쿨다운 무시, 즉시 발송
- 한국 공휴일(대체공휴일 포함)에 한국 주식 스캔이 실행되면? → 현재는 평일만 필터링하며, 공휴일 캘린더 연동은 향후 고도화
- 시스템 재시작 시 이전 신호 상태가 유실되면? → 마지막 신호 상태를 DB에 영속화하여 재시작 후에도 상태 전환 판정이 정확하게 동작
- 데이터 소스가 반환하는 가격 타임스탬프가 1시간 이상 오래된 경우? → 해당 데이터를 사용하지 않고 스킵 + 경고 로그
- 지표 계산 중 NaN 값이 발생하면? → NEUTRAL 유지, 신호 전환 없음으로 처리
- 텔레그램 봇 토큰이 설정되지 않은 상태에서 시스템을 시작하면? → 시스템은 정상 동작하되 알림 발송만 비활성화, 대시보드와 로그는 정상 동작
- 스캔 주기(10분) 내에 이전 스캔이 아직 완료되지 않았으면? → 새 스캔을 스킵하고 경고 로그 기록(중복 실행 방지)
- 차트 데이터 요청 시 캔들 데이터가 200개 미만이면? → 보유한 데이터만 표시하되, 일부 지표(EMA200 등)가 비어있을 수 있음을 차트에 안내
- yfinance 배치 다운로드에서 일부 종목만 실패하면? → 성공한 종목은 정상 처리, 실패 종목만 개별 fallback 시도
- 동일 종목이 다른 타임프레임으로 중복 등록되면? → (market, symbol) 유니크 제약으로 거부 — 타임프레임은 기존 항목에서 변경

## Requirements *(mandatory)*

### Functional Requirements

**신호 엔진**
- **FR-001**: 시스템은 볼린저밴드(20기간, 2시그마), RSI(14기간), MACD(12/26/9), EMA(20/50/200), 거래량(20일 평균 대비) 지표를 조합하여 매매 신호를 판정해야 한다.
- **FR-002**: 매수 신호는 4가지 필수 조건(BB 하단 터치/이탈 후 복귀, RSI < 30, MACD 히스토그램 상승 전환, 거래량 > 평균 1.2배) 모두 충족 시 발생해야 한다.
- **FR-003**: 매도 신호는 4가지 필수 조건(BB 상단 터치/이탈 후 복귀, RSI > 70, MACD 히스토그램 하락 전환, 거래량 > 평균 1.2배) 모두 충족 시 발생해야 한다.
- **FR-004**: 신호 강도(confidence)를 0~100 점수로 산정해야 한다 — 필수 조건 각 20점(총 80점), 스퀴즈 해제 +15점, EMA 정배열/역배열 +5점, RSI 극단치 최대 +10점.
- **FR-005**: 신호 등급을 점수 기반으로 분류해야 한다 — STRONG(90~100), NORMAL(70~89), WEAK(60~69), 미발생(60 미만).
- **FR-006**: NEUTRAL 복귀 조건을 판정해야 한다 — BUY→NEUTRAL: RSI > 50 AND 가격 > BB 중간선, 또는 MACD 하락 전환. SELL→NEUTRAL: RSI < 50 AND 가격 < BB 중간선, 또는 MACD 상승 전환.
- **FR-007**: BB 스퀴즈(볼린저밴드 vs 켈트너채널 비교)를 4단계(NO/LOW/MID/HIGH)로 판별해야 한다.

**데이터 수집**
- **FR-008**: 한국 주식은 종목번호 기반으로 시세 데이터를 수집하되, 1차 소스 실패 시 자동으로 2차, 3차 소스로 전환(fallback chain)해야 한다.
- **FR-009**: 미국 주식은 티커 기반, 암호화폐는 거래소 심볼 기반으로 시세 데이터를 수집하되, 각각 fallback 소스를 지원해야 한다.
- **FR-010**: 데이터 수집 실패 시 최대 3회 재시도(지수 백오프: 2초, 4초, 8초)를 수행해야 한다.
- **FR-011**: 지원 타임프레임은 15분, 30분, 1시간, 4시간, 일봉이며, 기본값은 1시간이어야 한다.
- **FR-011a**: 수집된 OHLCV 데이터를 종목/타임프레임별로 캐시하여 차트 API 호출 시 외부 소스 재호출 없이 응답해야 한다. 종목당 최대 500개 캔들을 유지하고 오래된 데이터는 자동 정리해야 한다.

**스케줄링**
- **FR-012**: 10분 간격 크론 스케줄로 워치리스트 전체를 자동 스캔해야 한다.
- **FR-013**: 시장별 거래 시간(한국 09:00~15:30 KST 평일, 미국 09:30~16:00 ET 평일, 비트코인 24시간)을 인식하여 장중 종목만 스캔해야 한다.
- **FR-014**: 미국 시장의 서머타임/동절기 시간대 전환을 자동 감지해야 한다.

**알림**
- **FR-015**: 신호 상태 전환(이전 상태 != 현재 상태) 시에만 텔레그램 알림을 발송해야 한다.
- **FR-016**: 알림 메시지에 종목명, 현재가, 등락률, 신호 방향, 강도 점수/등급, 지표 상세(BB/RSI/MACD/스퀴즈/거래량/EMA), 감지 시각, 타임프레임을 포함해야 한다.
- **FR-017**: 동일 종목 동일 방향 알림에 쿨다운(기본 30분)을 적용하되, 방향 전환(BUY↔SELL)은 쿨다운을 무시하고 즉시 발송해야 한다.
- **FR-018**: 최소 신호 등급 필터(WEAK/NORMAL/STRONG)를 설정할 수 있어야 한다.
- **FR-019**: 텔레그램 발송 실패 시 최대 3회 재시도하고, 실패 이력을 기록해야 한다.

**워치리스트**
- **FR-020**: 한국 주식(종목번호), 미국 주식(티커), 암호화폐(심볼) 3개 시장의 종목을 워치리스트에 추가/제거/활성화/비활성화할 수 있어야 한다.
- **FR-021**: 종목 추가 시 데이터 소스 조회를 통해 유효성을 검증하고, 종목명을 자동으로 가져와야 한다.
- **FR-022**: 종목별로 타임프레임을 개별 설정할 수 있어야 한다.

**대시보드**
- **FR-023**: 워치리스트 종목을 시장별로 그룹화한 카드 형태로 표시해야 한다 — 종목명, 현재가, 등락률, 신호 상태 배지, 신호 강도, 스퀴즈 단계, RSI/%B/BBW 미니 게이지, 마지막 전환 시각.
- **FR-024**: 신호 상태 변경 시 페이지 새로고침 없이 실시간으로 대시보드를 업데이트해야 한다.
- **FR-025**: 종목 상세 뷰에서 지표 현재 값, 신호 전환 이력 타임라인, 알림 발송 이력을 제공해야 한다.
- **FR-025a**: 종목 상세 뷰에서 로컬 OHLCV 데이터 기반 인터랙티브 캔들스틱 차트를 표시해야 한다. 차트에는 다음 지표가 오버레이/서브차트로 포함된다:
  - 메인 차트: 캔들스틱 + 볼린저밴드(상단/중간/하단 밴드) + EMA(20/50/200) 라인
  - 서브차트 1: RSI(14) 라인 + 과매수(70)/과매도(30) 수평선
  - 서브차트 2: MACD 라인 + 시그널 라인 + 히스토그램(양/음 색상 구분)
  - 서브차트 3: 거래량 바(양봉/음봉 색상 구분) + 20일 평균 거래량 라인
  - 스퀴즈 표시: MACD 서브차트 하단 또는 별도 행에 4단계 도트(NO/LOW/MID/HIGH 색상 구분)
- **FR-025b**: 차트에 신호 전환 시점을 BUY(녹색 상향 화살표)/SELL(적색 하향 화살표) 마커로 표시해야 한다.
- **FR-025c**: 차트는 줌, 스크롤, 크로스헤어(십자선) 인터랙션을 지원해야 하며, 타임프레임 전환(15m/30m/1h/4h/1d) 버튼을 제공해야 한다.
- **FR-026**: 시스템 헬스 정보(마지막 스캔 시각, 에러 카운트, 시장 개장/폐장 상태)를 표시해야 한다.

**웹훅**
- **FR-027**: TradingView Alert 웹훅을 수신하는 엔드포인트를 제공하고, 시크릿 기반 인증을 수행해야 한다.
- **FR-028**: 유효한 웹훅 데이터를 로컬 신호 파이프라인과 동일한 경로로 처리해야 한다.

**시스템 운영**
- **FR-029**: 모든 신호 전환 이력을 영속 저장하고 조회할 수 있어야 한다.
- **FR-030**: 시스템 에러(데이터 수집 실패, 알림 발송 실패 등)를 로그로 기록하고, 연속 3회 실패 시 텔레그램 시스템 에러 알림을 발송해야 한다.
- **FR-031**: 시스템 재시작 시 마지막 신호 상태를 복원하여 잘못된 중복 알림을 방지해야 한다.
- **FR-032**: 단일 프로세스에서 모든 기능(스케줄링, 데이터 수집, 지표 계산, 알림, 웹훅 수신, 대시보드)을 처리해야 한다.

### Key Entities

- **워치리스트 항목(Watchlist Item)**: 모니터링 대상 종목. 시장 구분(한국/미국/암호화폐), 심볼, 표시명, 분석 타임프레임, 데이터 소스, 활성화 여부를 가진다.
- **현재 신호(Current Signal)**: 각 워치리스트 종목의 최신 상태 캐시. 신호 상태(BUY/SELL/NEUTRAL), 강도 점수, 현재가, 각 지표 값(RSI, %B, BBW, 스퀴즈 레벨, MACD 히스토그램, 거래량 비율, EMA 값들), 마지막 업데이트 시각을 가진다. 워치리스트 항목과 1:1 관계.
- **신호 이력(Signal History)**: 모든 신호 판정 기록. 신호 상태, 이전 상태, 강도 점수, 타임프레임, 가격, 전체 지표 값, 감지 시각을 가진다. 워치리스트 항목과 1:N 관계.
- **알림 로그(Alert Log)**: 텔레그램 발송 기록. 발송 채널, 메시지 내용, 발송 시각, 성공 여부, 에러 메시지를 가진다. 신호 이력과 1:1 관계.
- **OHLCV 캐시(OHLCV Cache)**: 차트 렌더링용 캔들 데이터 캐시. 종목, 타임프레임, 타임스탬프, OHLCV 값을 가진다. 워치리스트 항목과 1:N 관계. 종목당 최대 500개 캔들 유지.
- **시스템 로그(System Log)**: 운영 이벤트 기록. 로그 레벨(INFO/WARN/ERROR), 소스(fetcher/indicator/telegram/scheduler), 메시지, 상세 정보, 발생 시각을 가진다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 매수/매도 조건 충족 후 최대 1회 스캔 주기(10분) + 처리 시간(1분 이내) 내에 텔레그램 알림이 사용자에게 도달한다.
- **SC-002**: 한국/미국/비트코인 3개 시장 종목을 하나의 시스템에서 통합 모니터링할 수 있다.
- **SC-003**: 1차 데이터 소스 장애 시 사용자 개입 없이 자동으로 대체 소스로 전환되며, 모니터링 중단 시간이 1회 스캔 주기(10분)를 초과하지 않는다.
- **SC-004**: 시스템 시작 후 사용자 개입 없이 시장별 거래 시간에 맞춰 24시간 자동 운영된다.
- **SC-005**: 대시보드에서 전체 워치리스트의 신호 현황을 10초 이내에 파악할 수 있다.
- **SC-006**: 신호 전환 알림의 오탈(false positive) 비율이 수동 차트 분석 대비 동등하거나 낮다 — 동일 지표 조합/파라미터 기준.
- **SC-007**: 시스템 설치부터 첫 알림 수신까지 30분 이내에 완료할 수 있다(환경 구성 + 종목 등록 + 첫 스캔).
- **SC-008**: 워치리스트 50개 종목을 1회 스캔 주기(10분) 내에 모두 처리할 수 있다.
- **SC-009**: 종목 상세 뷰의 지표 차트가 TradingView에서 동일 종목/타임프레임/지표 파라미터로 확인한 결과와 시각적으로 일치한다 (데이터 지연 차이 허용).
- **SC-010**: 차트 데이터(OHLCV + 지표) 로딩이 3초 이내에 완료되고, 줌/스크롤 인터랙션이 끊김 없이 동작한다.

## Assumptions

- 사용자는 1인 개인 투자자이며, 로컬 머신(macOS/Linux)에서 시스템을 운영한다.
- 인터넷 연결이 안정적으로 유지되며, 데이터 소스 API에 접근 가능하다.
- 텔레그램 봇은 사전에 생성되어 있고, 토큰과 채팅 ID를 사용자가 보유하고 있다.
- 한국 주식 데이터는 최대 15분 지연이 허용된다(무료 데이터 소스 한계).
- 기술적 지표의 파라미터(BB 20/2, RSI 14, MACD 12/26/9 등)는 TradingView 표준 기본값을 사용하며, 향후 커스텀 설정을 지원한다.
- 한국 공휴일 필터링은 초기 버전에서 미지원하며, 평일 기준으로만 장중 여부를 판별한다.
- 외부 인증/권한 관리는 불필요하다(로컬 전용 시스템).

## Scope Boundaries

**포함**:
- 기술적 지표 기반 매매 신호 판정 엔진
- 한국/미국/비트코인 3개 시장 데이터 수집 + fallback
- 텔레그램 알림 발송
- 워치리스트 CRUD
- 웹 대시보드(실시간 현황 + 신호 이력)
- 로컬 지표 차트 시각화 (캔들스틱 + BB/RSI/MACD/EMA/거래량/스퀴즈 오버레이 + 신호 마커)
- TradingView 웹훅 수신
- 시장시간 기반 자동 스케줄링
- 시스템 헬스 모니터링 + 에러 알림
- 다크모드 지원 (대시보드 UI)

**제외**:
- 자동 매매(주문 실행) 기능
- 백테스트/전략 시뮬레이션
- 멀티 유저/인증 시스템
- 모바일 앱 또는 PWA
- TradingView 스크리너 연동(종목 자동 발견)
- 한국 공휴일 캘린더 연동
- 클라우드 배포/Docker 지원
