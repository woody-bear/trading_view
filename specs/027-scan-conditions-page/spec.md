# Feature Specification: 조회조건 페이지

**Feature Branch**: `027-scan-conditions-page`  
**Created**: 2026-04-19  
**Status**: Draft  
**Input**: 추천종목·눌림목의 조회 조건을 확인하는 페이지 신설, 헤더 탭 추가

## Clarifications

### Session 2026-04-19

- Q: 조건 표시 형식 — 각 조건 블록의 내용을 어떤 시각적 형식으로 표현할 것인가? → A: Mermaid 스타일의 흐름도(플로우차트) 형태로 어떤 순서로 조회가 진행되는지 직관적으로 알 수 있도록 표시한다.
- Q: 코드 모듈화 범위 — 소스코드 모듈화 개선이 프론트엔드, 백엔드, 혹은 양쪽 모두에 해당하는가? → A: 프론트엔드 + 백엔드 각각 모듈화 (프론트: `conditions.ts` 상수 파일, 백엔드: 조건 상수/파이프라인 Python 모듈). 두 레이어가 각자 단일 소스를 가지며 독립적으로 참조된다.
- Q: 플로우차트 구조 — 5개 독립 플로우차트 vs 단일 통합 흐름? → A: 단일 통합 BUY 파이프라인 (BUY/SQZ BUY 라벨 판정이 진입점, 이후 데드크로스 제외 → 최근성 필터 → 추천종목 확정 → 눌림목 분기 → 눌림목 확정 순으로 좁혀나감). SELL은 매수 흐름과 무관하므로 별도 소형 플로우차트로 분리.
- Q: 모바일 화면 구성 전략 → A: 모바일에서는 플로우차트(도표)를 표시하지 않고, 동일한 조건 정보를 단계별 조건표(table/list)로 표시한다. PC는 Mermaid 통합 도표 + SELL 도표, 모바일은 조건표만 보여서 가독성 확보.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 매수 통합 파이프라인 확인 (PC: 도표 / 모바일: 조건표) (Priority: P1)

사용자가 차트의 BUY/SQZ BUY 라벨 발생부터 메인페이지 추천종목·눌림목 선정까지의 전체 흐름을 확인할 수 있다. 핵심조건(라벨 판정)이 진입점이고, 이후 조회조건 필터들이 차례로 결합되어 종목 후보를 좁혀나가는 과정이 표현된다. **PC에서는 단일 통합 Mermaid 플로우차트로, 모바일에서는 동일 정보를 단계별 조건표(table/list)로** 표시한다.

**Why this priority**: 매수 후보 선정의 전체 의사결정 흐름을 한눈에 파악하는 것이 본 페이지의 핵심 가치이며, 사용자가 추천종목/눌림목을 신뢰하는 근거가 된다.

**Independent Test**: PC(≥768px)에서는 통합 BUY 파이프라인 플로우차트가 단일 도표로, 모바일(<768px)에서는 동일 단계가 조건표로 표시되면 완성으로 간주한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 PC(≥768px) 화면에서 '조회조건' 페이지에 접근, **When** 페이지가 로드, **Then** 페이지 상단에 단일 통합 BUY 파이프라인 Mermaid 플로우차트가 표시된다.
2. **Given** 사용자가 모바일(<768px) 화면에서 '조회조건' 페이지에 접근, **When** 페이지가 로드, **Then** 도표 대신 단계별 조건표(또는 카드 리스트) 형태로 동일 정보가 위→아래 순서로 표시된다.
3. **Given** PC 통합 플로우차트 진입부, **When** 사용자가 흐름을 따라가면, **Then** 핵심조건(BUY 라벨 판정 분기와 SQZ BUY 라벨 판정 분기)이 맨 처음에 위치한다.
4. **Given** BUY 판정 분기 (PC 도표 또는 모바일 조건표 1단계), **When** 사용자가 확인하면, **Then** BB 하단 터치/돌파 → RSI 필터(프리셋별 30/35/40) → 모멘텀 상승 → BUY 라벨 발생 (또는 BB 하단 복귀 → RSI 필터 → BUY 라벨 발생) 조건이 표시된다.
5. **Given** SQZ BUY 판정 분기, **When** 사용자가 확인하면, **Then** 스퀴즈 해제 → 모멘텀 양수 → 모멘텀 상승 → SQZ BUY 라벨 발생 조건이 표시된다.
6. **Given** 라벨 발생 노드 이후, **When** 흐름을 이어가면, **Then** 데드크로스 제외 → 최근 20거래일 이내 신호 → 데이터 신선도 7일 이내 → 추천종목 확정 단계가 차례로 표시된다.
7. **Given** 추천종목 확정 노드, **When** 다음으로 이동하면, **Then** 분기점에서 '눌림목 필터(EMA20>EMA60>EMA120 + EMA5 하락)'가 통과되면 눌림목 확정 노드로 이어지고, 미통과 시 추천종목으로만 분류됨이 표시된다.
8. **Given** 거래량 필터 부재, **When** 사용자가 전체를 보면, **Then** "거래량 필터 미적용" 정보가 명시된다.
9. **Given** 쿨다운 규칙, **When** 라벨 발생 단계 근처에서 표시되면, **Then** "5봉 쿨다운" 조건이 포함된다.

### User Story 2 — SELL 라벨 별도 표시 (PC: 도표 / 모바일: 조건표) (Priority: P2)

사용자가 차트의 SELL 라벨 발생 조건을 별도 영역에서 확인할 수 있다. SELL은 추천종목/눌림목 선정에 사용되지 않고 차트 표시 전용이므로 매수 영역과 분리되어 표시된다. **PC에서는 별도 Mermaid 플로우차트로, 모바일에서는 조건표로** 표시한다.

**Why this priority**: 매도 신호도 투자 판단에 중요하지만, 메인 매수 파이프라인과 분리되므로 우선순위는 P2.

**Independent Test**: PC에서는 SELL 별도 플로우차트, 모바일에서는 SELL 조건표가 매수 영역 하단에 표시되고, BB 상단 → RSI > 60 → 모멘텀 하락 → SELL 라벨 흐름과 안내 문구가 명확히 표현되면 완성으로 간주한다.

**Acceptance Scenarios**:

1. **Given** PC 화면, **When** 사용자가 매수 영역 아래로 스크롤, **Then** SELL 라벨 별도 Mermaid 플로우차트가 표시된다.
2. **Given** 모바일 화면, **When** 사용자가 매수 조건표 아래로 스크롤, **Then** SELL 조건표가 표시된다 (도표 미표시).
3. **Given** SELL 표현(PC 도표 또는 모바일 조건표), **When** 사용자가 확인하면, **Then** BB 상단 터치/돌파 → RSI > 60(고정값) → 모멘텀 하락 → SELL 라벨 발생 (또는 BB 상단 복귀 → RSI > 60 → SELL 라벨 발생) 조건이 표시된다.
4. **Given** SELL 영역, **When** 표시 정보를 보면, **Then** "SELL 라벨은 차트 마커 표시 전용이며 추천종목/눌림목 선정에 사용되지 않음"이 안내된다.

### User Story 3 — 조건 정의 코드 모듈화 (프론트 + 백엔드) (Priority: P2)

프론트엔드와 백엔드 각각에서 조건 판정 로직을 재사용 가능한 단일 모듈로 분리한다. 프론트는 `conditions.ts` 상수 파일로 통합 BUY 파이프라인과 SELL 플로우차트의 노드/엣지 데이터를 중앙화하고, 백엔드는 `scan_conditions.py` 모듈로 조건 상수와 필터 함수를 추출해 스캔 로직이 참조하도록 한다.

**Why this priority**: 현재 조건 로직이 `full_market_scanner.py` 내부에 하드코딩되어 있어 변경과 테스트가 어렵다. 각 레이어를 독립적으로 모듈화하면 유지보수성과 테스트 용이성이 향상된다.

**Independent Test**: 프론트 `conditions.ts`에서 플로우차트 노드 라벨을 변경하면 페이지에 즉시 반영되고, 백엔드 `scan_conditions.py`에서 쿨다운 값을 변경하면 스캔 로직에 반영되면 완성으로 간주한다.

**Acceptance Scenarios**:

1. **Given** 프론트 `conditions.ts`, **When** 통합 파이프라인 노드 라벨을 변경, **Then** 조회조건 페이지에 즉시 반영된다.
2. **Given** 백엔드 `scan_conditions.py`, **When** 신호 탐색 범위(20거래일)를 수정, **Then** `full_market_scanner.py` 수정 없이 스캔 로직에 반영된다.
3. **Given** 리팩토링된 백엔드 모듈, **When** 기존 스캔 기능을 실행, **Then** 리팩토링 전과 동일한 결과가 생성된다.

### User Story 4 — 헤더 탭 네비게이션 (Priority: P3)

사용자가 현재 어느 페이지에 있든 헤더의 '조회조건' 탭을 클릭하여 해당 페이지로 즉시 이동할 수 있다.

**Why this priority**: 탭 네비게이션은 페이지 접근성의 기반이나, 콘텐츠보다 우선순위는 낮다.

**Independent Test**: 헤더에 '조회조건' 탭이 표시되고 클릭 시 해당 페이지로 이동하면 완성으로 간주한다.

**Acceptance Scenarios**:

1. **Given** 메인 페이지, **When** 헤더에서 '조회조건' 탭 클릭, **Then** 조회조건 페이지로 이동하고 탭이 활성화 상태로 표시된다.
2. **Given** 조회조건 페이지, **When** 헤더에서 다른 탭 클릭, **Then** 해당 페이지로 이동한다.
3. **Given** 모바일 화면, **When** 하단 네비게이션에서 조회조건 접근, **Then** 동일하게 페이지 이동이 동작한다.

---

### Edge Cases

- 민감도 설정이 변경된 경우: RSI 임계값 표시는 현재 설정값이 아닌 프리셋 전체(strict/normal/sensitive별 값)를 통합 파이프라인의 분기 노드(PC) 또는 조건표 행(모바일)으로 나열한다.
- 향후 조건이 변경될 경우: 모듈화된 조건 소스(프론트 `conditions.ts` / 백엔드 `scan_conditions.py`)를 수정하면 PC 도표·모바일 조건표·스캔 로직이 동시에 업데이트되어야 한다.
- 모바일에서는 Mermaid 도표를 렌더링하지 않으므로 mermaid 라이브러리 동적 import는 PC 폭(≥768px) 진입 시에만 트리거되어 모바일 페이지 로드 비용을 절감한다.
- 화면 폭이 PC↔모바일 경계(768px)에서 동적으로 변경될 경우: 새로고침 또는 반응형 분기로 도표/조건표 표시가 전환되어야 한다.
- BUY와 SQZ BUY 라벨 판정 분기는 진입부에서 두 경로로 갈라진 뒤 라벨 발생 노드 이후 동일한 후속 필터 단계로 합쳐져 표현된다(PC 도표는 merge 노드, 모바일 조건표는 "공통 후속 단계" 섹션으로 통합).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 헤더 탭 목록에 '조회조건' 항목을 추가해야 한다.
- **FR-002**: '조회조건' 탭 클릭 시 조회조건 전용 페이지로 이동해야 한다.
- **FR-003**: 조회조건 페이지는 (a) 매수 통합 파이프라인 영역과 (b) SELL 별도 영역 두 부분으로 구성되어야 한다.
- **FR-003a**: 화면 폭이 768px 이상(PC)일 때는 두 영역 모두 Mermaid 플로우차트로 표시한다.
- **FR-003b**: 화면 폭이 768px 미만(모바일)일 때는 Mermaid 도표 대신 동일 정보를 단계별 조건표(table 또는 카드 리스트)로 표시한다. 두 영역 모두 동일하게 적용된다.
- **FR-004**: 매수 통합 파이프라인은 핵심조건(BUY/SQZ BUY 라벨 판정)이 진입점이며 이후 조회조건 필터가 결합되어 추천종목 → 눌림목으로 좁혀가는 흐름을 표현해야 한다 (PC: 단일 플로우차트, 모바일: 단계별 조건표 위→아래 순서).
- **FR-005**: 통합 파이프라인 진입부의 BUY 판정 분기는 다음을 포함해야 한다: BB 하단 터치/돌파 → RSI 필터(프리셋별 30/35/40) → 모멘텀 상승 → BUY 라벨, 또는 BB 하단 복귀 → RSI 필터 → BUY 라벨.
- **FR-006**: 통합 파이프라인 진입부의 SQZ BUY 판정 분기는 다음을 포함해야 한다: 스퀴즈 해제(전봉 ON → 현봉 OFF) → 모멘텀 양수 → 모멘텀 상승 → SQZ BUY 라벨.
- **FR-007**: 라벨 발생 노드 이후의 통합 파이프라인은 다음 필터 단계를 순서대로 포함해야 한다: (1) EMA 5선 전체 역배열(EMA5 < EMA10 < EMA20 < EMA60 < EMA120) → 제외, (2) 최근 20거래일 이내 BUY/SQZ BUY 신호 → 없으면 제외, (3) 데이터 신선도 7일 이내 → 초과 시 제외, (4) 거래량 필터 미적용(통과) → 추천종목 확정.
- **FR-008**: 추천종목 확정 노드 이후 분기로 눌림목 필터를 표현해야 한다: EMA20 > EMA60 > EMA120(장기 상승추세) AND EMA5 현재 < 직전(단기 눌림) → 통과 시 눌림목 확정, 미통과 시 추천종목으로만 분류.
- **FR-009**: 통합 파이프라인은 쿨다운 5봉 규칙을 라벨 발생 노드 근처에 시각적으로 표시해야 한다.
- **FR-010**: SELL 별도 영역은 다음 흐름을 포함해야 한다: BB 상단 터치/돌파 → RSI > 60(고정값) → 모멘텀 하락 → SELL 라벨, 또는 BB 상단 복귀 → RSI > 60 → SELL 라벨, 쿨다운 5봉 규칙 포함 (PC: 별도 플로우차트, 모바일: 조건표).
- **FR-010a**: SELL 영역에는 "SELL 라벨은 차트 표시 전용이며 추천종목/눌림목 선정에 사용되지 않음" 안내 문구가 표시되어야 한다.
- **FR-011**: 조건 판정 로직의 프론트엔드 측 데이터(통합 파이프라인·SELL 영역의 PC 도표 정의 + 모바일 조건표 단계 정의, 임계값, 라벨 텍스트)는 재사용 가능한 단일 모듈(예: `frontend/src/constants/conditions.ts`)에서 관리되어야 한다. PC 도표용 Mermaid DSL과 모바일 조건표용 단계 데이터는 동일한 의미적 단계 정의에서 파생되어야 한다.
- **FR-011a**: 백엔드 측 조건 상수(RSI 임계값, 쿨다운 봉 수, EMA 정/역배열 판정, 신호 탐색 기간 등)와 파이프라인 필터 함수는 재사용 가능한 단일 모듈(예: `backend/services/scan_conditions.py`)로 추출되어야 한다. 기존 `full_market_scanner.py`가 이 모듈을 참조하도록 리팩토링한다.
- **FR-011b**: 프론트엔드와 백엔드 모듈은 각자 독립적인 단일 소스를 가지며 직접 연동하지 않는다. 조건 값 변경 시 두 레이어 각각에서 한 곳씩 수정한다.
- **FR-012**: 페이지는 화면 폭에 따라 자동으로 표현 방식을 전환한다 — PC(≥768px): Mermaid 플로우차트 표시, 모바일(<768px): Mermaid 도표를 표시하지 않고 조건표만 표시. 모바일 조건표는 가로 스크롤 없이 세로 스크롤만으로 전체 정보를 확인할 수 있어야 한다.
- **FR-013**: 현재 활성화된 탭이 헤더에서 시각적으로 구분되어야 한다.

### Key Entities

- **조건 단계(Step)**: 조건 판정 파이프라인의 각 단계를 표현하는 의미적 단위. 속성: 단계 ID, 단계명, 조건 설명, 통과/제외 결과, 단계 유형(진입/조건/분기/종료). 이 단일 정의가 PC 도표와 모바일 조건표 양쪽의 소스가 된다.
- **플로우차트 노드/엣지**: PC에서 Mermaid가 렌더링하는 시각적 단위. 조건 단계 데이터에서 자동 또는 수동으로 매핑된다.
- **모바일 조건표 행/카드**: 모바일에서 표시되는 단계별 항목. 동일한 조건 단계 데이터에서 파생된다.
- **통합 BUY 파이프라인**: 라벨 판정(핵심조건) → 필터(조회조건) → 추천종목 → 눌림목 분기까지를 하나의 흐름으로 묶은 표현. PC는 단일 Mermaid 플로우차트, 모바일은 단계별 조건표.
- **SELL 별도 영역**: 매수 흐름과 분리된 SELL 라벨 발생 조건. PC는 별도 Mermaid 플로우차트, 모바일은 SELL 조건표.
- **프론트엔드 조건 모듈**: 조건 단계(Step) 정의, PC용 Mermaid DSL, 임계값, 라벨 텍스트를 담은 상수 파일. 조회조건 페이지가 참조한다.
- **백엔드 조건 모듈**: RSI 임계값, 쿨다운, EMA 정/역배열 판정, 파이프라인 필터 함수를 담은 Python 모듈. 스캔 로직이 참조한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: '조회조건' 탭 클릭 후 페이지 로드까지 1초 이내에 완료된다.
- **SC-002**: PC(≥768px)에서 페이지 상단에 매수 통합 파이프라인 단일 Mermaid 플로우차트가 표시되고, 진입부 라벨 판정(BUY/SQZ BUY)부터 눌림목 확정까지 최소 8개 이상의 노드와 분기 1회 이상이 포함된다.
- **SC-002a**: 모바일(<768px)에서 페이지 상단에 매수 통합 파이프라인 단계별 조건표가 표시되고, 동일 8단계 이상이 위→아래 순서로 나열되며 가로 스크롤 없이 세로 스크롤만으로 전체 확인 가능하다.
- **SC-003**: PC에서 SELL 별도 플로우차트가 표시되고, 최소 3개 이상의 노드 + 안내 문구가 포함된다.
- **SC-003a**: 모바일에서 SELL 조건표가 표시되고, 최소 3단계 + 안내 문구가 포함된다.
- **SC-004**: 모바일(375px 기준) 및 PC(1280px 기준) 화면 모두에서 텍스트 잘림 없이 매수·SELL 두 영역의 전체 정보가 표시된다.
- **SC-005**: 헤더 탭에서 '조회조건' 클릭 시 100% 성공률로 해당 페이지로 이동한다.
- **SC-006**: 프론트엔드 조건 값 변경 시 `conditions.ts` 단일 파일만 수정하면 PC 도표와 모바일 조건표 양쪽에 반영된다.
- **SC-007**: 백엔드 조건 값 변경 시 `scan_conditions.py` 단일 모듈만 수정하면 스캔 로직에 반영된다(`full_market_scanner.py` 수정 불필요).

## Assumptions

- PC용 플로우차트는 Mermaid 라이브러리로 구현한다. 모바일에서는 mermaid 라이브러리를 로드하지 않는다(번들 크기·로드 비용 절감).
- 모바일 조건표는 일반 HTML/CSS(테이블 또는 카드 리스트)로 구현하며 별도 라이브러리 의존성 없음.
- PC↔모바일 분기 기준은 화면 폭 768px (Tailwind `md` breakpoint).
- 민감도 프리셋 RSI 값은 세 개 모두(strict=30, normal=35, sensitive=40)를 도표 분기 노드(PC) 또는 조건표 행(모바일)으로 표현한다.
- 헤더 탭 추가 위치는 기존 헤더 탭 구조의 마지막(또는 사용성상 적절한 위치)으로 한다.
- 모바일 하단 네비게이션(BottomNav)에도 동일하게 '조회조건' 항목이 추가된다.
- 코드 모듈화: 프론트(`conditions.ts`) + 백엔드(`scan_conditions.py`) 각자 독립 단일 소스. 레이어 간 직접 연동 없음.
- BUY와 SQZ BUY 진입부 분기는 라벨 발생 단계 이후 단일 후속 필터 흐름으로 merge한다 (PC: merge 노드, 모바일: "공통 후속 단계" 섹션).
- 프론트엔드 `conditions.ts` 내 단계(Step) 정의는 PC Mermaid DSL과 모바일 조건표 양쪽 렌더링의 단일 소스 역할을 한다.
